= Viewing / View
:toc:

`viewing/view` is the Smart Responsor component responsible for the central Symfony view boundary.

It keeps producer components from rendering templates directly. Producer controllers return a neutral View Payload. Viewing decides whether the final response is structured JSON or HTML rendered through a template fallback chain.

== Canon

* Repository/package: `viewing/view`
* Namespace: `App\Viewing`
* Business prefix: `View*`
* Bundle class: `App\Viewing\ViewingBundle`
* Twig namespace: `@Viewing`
* Main event boundary: `kernel.view`
* Defensive event boundary: `kernel.response`
* Producer components: data producers only
* Interfacing: passive template/layout provider only
* Viewing: active view/render decision layer

== Main rule

Producer controllers must not call Twig, `render()`, `renderView()`, custom template providers, render adapters, or arbitrary template lookups.

They return a neutral payload:

[source,php]
----
return [
    '_view' => [
        'surface' => 'vendor',
        'operation' => 'show',
        'intent' => 'profile',
        'format' => 'auto',
        'component' => 'Cruding',
    ],
    'locations' => [
        'body' => [
            'entity' => $vendorPayload,
        ],
    ],
    'meta' => [
        'title' => 'Vendor profile',
    ],
];
----

Viewing converts this into either a Twig HTML response or a structured JSON fallback.

== Exact fallback chain

For a human HTML request, Viewing attempts templates in this exact order. The word `templates` means the package filesystem template root, not an additional Twig path segment.

[source,text]
----
1. Interfacing/templates/<resource>/index.html.twig
   Twig: @Interfacing/<resource>/index.html.twig

2. Interfacing/templates/index.html.twig
   Twig: @Interfacing/index.html.twig

3. <Component>/templates/index.html.twig
   Twig: @<Component>/index.html.twig

4. Viewing/templates/view/index.html.twig
   Twig: @Viewing/view/index.html.twig

5. Structured JSON fallback
----

The local component fallback is rendered by Viewing, not by the producer controller. Controllers do not regain rendering control.

== Viewing processes itself

Viewing is both a connectable bundle and a standalone development application shell. Its own controllers follow the same rule:

[source,text]
----
Viewing controller returns View Payload
-> ViewKernelViewSubscriber catches it
-> Interfacing/view/index if present
-> Interfacing/index if present
-> Viewing/templates/index.html.twig
-> Viewing diagnostic template
-> JSON fallback
----

The included `/viewing` controller proves the rule: it returns data only and never renders directly.

== Bundle installation sketch

[source,php]
----
// config/bundles.php
return [
    App\Viewing\ViewingBundle::class => ['all' => true],
];
----

Optional route import for the Viewing diagnostic/home route:

[source,php]
----
// config/routes/viewing.php
return static function (Symfony\Component\Routing\Loader\Configurator\RoutingConfigurator $routes): void {
    $routes->import('@ViewingBundle/Resources/config/routes.php');
};
----

Optional route defaults for guarded UI routes:

[source,yaml]
----
_view_controlled: true
----

== Standalone development shell

The repository contains a small Symfony shell for local checks:

[source,text]
----
config/bundles.php
config/packages/framework.php
config/packages/twig.php
config/packages/viewing.php
config/routes/viewing.php
public/index.php
bin/console
src/Kernel.php
----

Standalone mode is not a different architecture. It uses the same bundle, the same route file, and the same `kernel.view` subscriber.

== Bot/JSON policy

A request marked as bot traffic should not trigger template lookup. Viewing immediately returns structured JSON.

The current implementation reads the request attribute `_view_actor_type`; values such as `bot` force JSON.

== Defensive response guard

`kernel.response` cannot reconstruct lost controller data after an illegal direct render. It is only a drift-containment guard.

It can replace illegal HTML responses on explicitly controlled routes with a structured JSON problem response. Controlled routes should set `_view_controlled: true` or equivalent configuration in the host/application route defaults.

== Wave 4 hardening

Wave 4 makes the component safer to connect into a host application:

[source,text]
----
ViewRouteExclusionService
ViewTrafficRequestSubscriber
ViewTrafficClassifier
production-safe diagnostic mode
viewing:drift:check console command
route/path exclusions for profiler, assets, health, metrics, robots, sitemap
stronger response guard integration
----

The traffic classifier is intentionally small. It marks obvious automated traffic with the configured actor request attribute before `kernel.view`. The render decision remains centralized in Viewing. Bot traffic receives JSON and does not trigger template lookup.

The drift checker is a guardrail for producer components. It scans controller directories for direct rendering, Twig injection, direct `.html.twig` references, and custom render/provider/adaptor drift.

[source,shell]
----
php bin/console viewing:drift:check src/Controller
php bin/console viewing:drift:check ../Cruding/src/Controller
----

A failing drift check means the producer is trying to regain rendering control and must be migrated back to View Payload output.
